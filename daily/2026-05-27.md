## 核心概念

### 1. 为什么 Agent 不能直接拿 EOA 私钥

EOA 私钥的问题不是“能不能自动执行”，而是**权限太大且不可细分**。

- 一旦 Agent 持有 EOA 私钥，它就默认拥有该账户的全部控制权
- 它不能天然限制“只能调哪个合约、只能调哪个方法、最多花多少钱、多久失效”
- 一旦被 Prompt Injection、工具滥用或密钥泄露击中，损失边界几乎等于账户本身的资产边界

所以问题本质不是“让 Agent 会签名”，而是：  
**怎么让 Agent 只在一小块、可撤销、可过期、可计量的权限盒子里签名。**

### 2. Session Key 的作用

Session Key 不是替代 Owner，而是给 Agent 一把**临时且受限的钥匙**。

它解决的是 5 个维度的问题：

- **合约范围**：只能调哪些合约
- **方法范围**：只能调用哪些函数
- **金额边界**：单笔和累计最多能动多少资产
- **时间边界**：这个权限多久有效
- **撤销机制**：失控后怎么停

### 3. AI Security 在这里防的不是“模型答错”，而是“模型做错事”

AI × Web3 里，风险不是停留在文本输出层，而会穿透到工具调用层。

典型链路是：

```text
恶意输入 / Prompt Injection
  ↓
模型被诱导生成错误操作意图
  ↓
工具层未做权限收口
  ↓
钱包签名 / 合约调用被执行
  ↓
链上动作不可逆
```

所以真正的安全边界不能只放在 Prompt 上，而要放在**钱包权限模型和工具执行约束**上。

## 最小场景：Sepolia 上的受限 ERC-20 转账 Agent

### 场景定义

让一个 Agent 帮我在 Sepolia 上执行 ERC-20 转账，但只允许它完成**小额、短时、单一用途**的操作。

### Session Key 权限边界表

| 维度 | 规则 |
|---|---|
| 链范围 | 仅限 **Sepolia** |
| 合约范围 | 仅限指定 ERC-20 合约地址 |
| 方法范围 | 仅允许 `transfer(address to, uint256 value)` |
| 明确禁止 | `approve`、`transferFrom`、任意未知合约调用、原生 ETH 转出 |
| 单笔限额 | ≤ 5 USDC |
| 累计限额 | 24 小时内 ≤ 20 USDC |
| 接收方限制 | 仅允许转给预先白名单地址 |
| 有效期 | 2 小时自动过期 |
| 使用主体 | 仅限当前 Agent 会话 |
| 撤销方式 | 到期自动失效；必要时由 owner 发起撤销交易，立即吊销 |

---

## 风险差异：`transfer` vs `approve` vs 任意合约调用

### `transfer`
风险相对最可控。

- 每次转出是显式动作
- 可直接配单笔限额和累计限额
- 泄露后，攻击者最多在权限窗口内转走被允许的那部分额度

### `approve`
风险通常比 `transfer` 更大。

- `approve` 不是当场花钱，而是给别人“以后从你账户里扣钱”的权力
- 如果批准额度过大，后续真正扣款可能发生在另一个交易里
- 表面看只是一次授权，实际可能把未来资产暴露给第三方合约

所以对 Agent 来说，**很多情况下宁可允许小额 `transfer`，也不要轻易开放 `approve`**。

### 任意合约调用
风险最大。

- 模型可能被诱导去调用一个看起来正常、实际恶意的合约
- 也可能在复杂合约里走入你没预料到的状态路径
- 一旦允许“任意目标 + 任意 calldata”，Session Key 的意义基本就被掏空了

---

## 今日收获

今天最重要的理解是：

**AI Agent 上链的关键不是“让它能签名”，而是“让它只能在可接受的损失范围内签名”。**

EOA 私钥的问题不是自动化能力不够，而是权限过于完整、泄露后不可收回；Session Key 的价值在于把权限拆细成“特定合约 + 特定方法 + 特定额度 + 特定时效”。这样即使 Agent 被 Prompt Injection 诱导，真正能穿透到链上的动作也仍然被收在一个小盒子里。

换句话说：

- AA 让账户变得可编程
- Session Key 让这种可编程性可以安全地下放给 Agent
- AI Security 真正落地的位置，不在嘴上说“别乱调工具”，而在权限模型本身
