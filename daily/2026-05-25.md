## 概念回顾

### Indexing 数据流

```
链上合约 emit Transfer(from, to, amount)
  ↓
区块打包，事件写入 receipt.logs[]
  ↓
eth_getLogs 按 address + topic 过滤拉取
  ↓
ABI 解码（topics[1]=from, topics[2]=to, data=amount）
  ↓
写入本地 DB / 输出结构化数据
  ↓
前端 / AI Agent 查询
```

### ERC-20 Transfer 事件结构

```solidity
event Transfer(address indexed from, address indexed to, uint256 value);
```

- **topic[0]**：事件签名 keccak256 hash = `0xddf252ad1be2c89b69c2b068fc378daa952ba7f163c4a11628f55a4df523b3ef`
- **topic[1]**：`from` 地址（indexed，32 字节，前 12 字节补零）
- **topic[2]**：`to` 地址（indexed，同上）
- **data**：`value`（非 indexed，abi 编码的 uint256，32 字节）

### RPC vs Indexing 的区别

| | RPC (`eth_getLogs`) | 索引服务 (The Graph / Etherscan API) |
|---|---|---|
| 数据新鲜度 | 实时 | 有延迟（秒到分钟） |
| 历史范围 | 受 node 限制，公共节点通常 ≤ 10,000 块/次 | 全历史 |
| 可靠性 | rate limit / 不处理 reorg | 通常已处理 reorg |
| 适合场景 | 学习、最小实践、实时监听 | 生产级查询 |

## 实践记录

### 实践目标

用 `eth_getLogs` 拉取 Sepolia 上 USDC 测试代币最近 200 个区块的 Transfer 事件，解码并打印 from / to / value / txHash / blockNumber。

### 操作步骤

1. **选择目标合约**：Sepolia USDC (Circle 官方测试版) `0x1c7D4B196Cb0C7B01d743Fbc6116a902379C7238`
2. **构造 eth_getLogs 请求**：
   - `address`: 合约地址
   - `topics[0]`: Transfer 事件签名 `0xddf252ad...`（所有 ERC-20 都一样）
   - `fromBlock` / `toBlock`: 最近 200 个区块
3. **解码 log 结构**：
   - `topics[1]` → from 地址（去掉左侧填充的 12 字节零）
   - `topics[2]` → to 地址
   - `data` → value（uint256 hex → BigInt → 除以 10^decimals）
4. **输出结构化 JSON**：写入 `01-output.json`，包含 meta 信息和所有事件

### 实践结果

**扫描范围**：区块 10,919,546 → 10,919,746（200 个区块）  
**捕获事件**：864 个 Transfer 事件  
**输出文件**：`experiments/indexing/01-output.json`（约 3400 行）

**前 5 条事件示例**（已解码）：

```json
[
  {
    "blockNumber": 10919546,
    "txHash": "0xe1a37993b60a2d3abe4d707164e78714130d6be4e6db50f7be507bf5b5ca6cc1",
    "from": "0x6ac9abf56d8bc88232d9bb551c18baadaa5fa5d3",
    "to": "0x9bfdff86f1460e38c7046a39b24a963ed623b3c5",
    "value": "0.000001 USDC"
  },
  {
    "blockNumber": 10919546,
    "txHash": "0xd8bffa7a82c463bf571a425971afc8dd0783a1d077e7a9e0a8fec45da92752cf",
    "from": "0x5c99bd9ab8b7da3f6d0a00216ca2905bc3e3bf86",
    "to": "0xd1ef43e63308ecebdb09865b5298db5d600ae44f",
    "value": "16.200805 USDC"
  }

```

### 关键发现

1. **200 区块 = 864 事件**：Sepolia USDC 很活跃，平均每区块 4+ 个 Transfer
2. **同一 txHash 可能有多个 Transfer**：例如 `0xd8bffa7a82...` 里有 logIndex 294 和 297 两条——这是合约内部调用链（swap / 路由 / 多跳）
3. **金额范围极大**：从 0.000001 USDC（测试粉尘）到 16+ USDC（真实转账）
4. **解码后的数据才是 AI Context**：原始 `data: "0x00000000000000000000000000000000000000000000000000000000000f6f85"` 对 LLM 无意义，解码成 `"value": "16.200805 USDC"` 才可用

## 卡点 / 疑问

1. **公共 RPC 的 block range 限制**：很多节点限制单次查询 ≤ 10,000 块，生产环境需要分批扫描 + 记录 checkpoint
2. **reorg 怎么处理**？脚本没处理链重组——如果扫描到的区块被 reorg，已索引的数据会失效。生产级 indexer 需要监听 `eth_subscribe('newHeads')` 并回滚
3. **timestamp 哪里来**？log 本身没有时间戳，需要额外调用 `eth_getBlockByNumber` 拿 `block.timestamp`
4. **如何增量扫描**？当前是一次性扫 200 块，实际应该是：上次扫到 N，这次从 N+1 开始

## 今日收获

### 核心认知

**Indexing 的本质**：把链上「不可查询的事件流」变成「可查询的结构化表」。

- **链上**：合约 `emit Transfer(from, to, value)` → 写入 `receipt.logs[]` → 永久存在，但只能按区块/交易查
- **索引层**：`eth_getLogs` 批量拉 → ABI 解码 → 写 DB（按 from / to / blockNumber / value 建索引）→ 前端/AI 可以查「某地址的所有转账历史」

### eth_getLogs 的三个关键参数

1. **address**：过滤合约地址（不填 = 扫所有合约，会超时）
2. **topics[0]**：事件签名（Transfer / Approval / Swap 等），是 keccak256 hash
3. **fromBlock / toBlock**：区块范围（太大会被 rate limit）

### 哪些字段该进 AI Context

✅ **该进**：`from`, `to`, `value`（人类可读格式）, `blockNumber`, `txHash`  
❌ **不该进**：`topics` 原始 hex, `data` 原始 hex, `logIndex`（除非调试）

**原因**：LLM 的 token 成本和推理质量取决于输入的「信息密度」。`0x00000...0f6f85` 占 66 字符但信息为零，`"16.200805 USDC"` 只占 14 字符但语义完整。

### Day 6 vs Day 8 的区别

| | Day 6 (eth_getTransactionByHash) | Day 8 (eth_getLogs) |
|---|---|---|
| 查询模式 | 单笔精确查询（需要知道 hash） | 批量扫描 + 过滤 |
| 适用场景 | 用户点击「查看交易详情」 | 构建「地址历史」/「代币流向」 |
| 是否是 Indexing | ❌ 不是 | ✅ 是 |
| AI Agent 用途 | 解释单笔交易 | 提供历史上下文（"你上周转了多少 USDC"） |
