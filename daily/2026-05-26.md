## 概念准备

### 5/24 概览卡片回顾

Account Abstraction 把账户的验证逻辑、付 gas 方式、权限策略从固定的 EOA 里解放出来，让账户变成可编程合约。

**核心组件**：
- **ERC-4337**：用 UserOperation + EntryPoint + Bundler，不改共识层
- **Smart Account**：合约控制的账户（多签 / 社交恢复 / 批量交易）
- **Bundler**：收集 UserOperation 并提交到 EntryPoint
- **Paymaster**：替用户付 gas
- **Session Key**：给应用或 Agent 的临时受限权限

### 5/24 遗留的卡点（今天要回答的）

## EOA 交易 vs UserOperation 对比

> 精读后填写

| 字段 | EOA Transaction | UserOperation (ERC-4337) |
|---|---|---|
| 发起者 | 用户| 用户或应用|
| 签名验证 | 协议层固定验证（ecrecover 从签名恢复地址，与 from 对比）| 智能账户的 validateUserOp() 函数（可编程，支持多签等自定义逻辑）|
| gas 支付 | 原生代币| 原生或非原生资产|
| 进入的 mempool | mempool| alt mempool|
| 执行路径 | 用户发起交易请求、交易经钱包确认、签名、广播、打包和执行，人工操作| 用户创建 UserOperation → Bundler 收集打包 → 提交到 EntryPoint 合约 → EntryPoint 调用智能账户 validateUserOp() 验证 → 验证通过后 EntryPoint 调用智能账户执行目标动作（Paymaster 可赞助 gas），可自动执行|
| 可编程性 | 较低| 较高|

## ERC-4337 数据流

> 精读后画出

```
用户/Agent 构造 UserOperation
  ↓
Bundler 收集 UserOp（通过 alt mempool）
  ↓  Bundler 链下模拟（simulation）预判是否能通过
  ↓
Bundler 将多个 UserOps 打包成一笔普通交易，提交到 EntryPoint 合约
  ↓
EntryPoint 调用 Smart Account 的 validateUserOp() —— 链上验证签名、nonce、策略
  ↓  （可选）EntryPoint 调用 Paymaster 的 validatePaymasterUserOp() 确认 gas 赞助
  ↓
验证通过 → EntryPoint 调用 Smart Account 执行目标操作（callData）
```

## 实践记录

### 工具选择：permissionless.js (Pimlico)

**选择原因**：
- 基于 viem，轻量，文档清晰
- Pimlico 提供免费的 Sepolia Bundler + Paymaster（测试网免费赞助 gas）
- 跟后续学 Session Key 衔接好

### 环境准备

1. 注册 [Pimlico Dashboard](https://dashboard.pimlico.io)，创建项目，获取 API Key
2. 创建实验目录并安装依赖：
   ```bash
   cd experiments && mkdir account-abstraction && cd account-abstraction
   npm init -y
   npm install permissionless viem
   ```
3. 创建 `.env` 文件（已在 .gitignore 中，不会被提交）：
   ```
   PIMLICO_API_KEY=你的key
   ```

### 实践 1 + 2：创建 Smart Account + 发送 UserOperation

**实现思路**：一个脚本完成全流程——生成 owner EOA → 创建 Smart Account → 发送 UserOperation（给自己转 0 ETH，验证通路）

**核心步骤**：
1. 用 `generatePrivateKey()` 生成一个临时 EOA 作为 owner（只用于签名，不需要有余额）
2. 用 `toSimpleSmartAccount()` 创建 Smart Account（此时合约还没部署，地址是预计算的）
3. 用 `createSmartAccountClient()` 连接 Pimlico Bundler + Paymaster
4. 调用 `sendTransaction()` 发送第一笔 UserOp（这笔交易同时部署 Smart Account 合约）

### 实践结果

**Owner EOA**：`0xf9AD51b29838eC76CA83F6544927cdDBec699afA`
**Smart Account**：`0x53cfaFF9c7Bea895A6aeeB97229df486b5126bC1`
**Transaction Hash**：`0xf3d661f7a51458730f3bb50dcf93b11631b4e6f8e0dfd06a643bc4fc516701d4`
**Etherscan**：https://sepolia.etherscan.io/tx/0xf3d661f7a51458730f3bb50dcf93b11631b4e6f8e0dfd06a643bc4fc516701d4

### 链上验证（用 RPC 查了交易详情）

| 观察点 | 实际数据 | 说明 |
|---|---|---|
| 交易 `from` | `0x4337002c5702ce424cb62a56ca038e31e1d4a93d` | **不是我的 EOA**，是 Pimlico 的 Bundler 地址 |
| 交易 `to` | `0x0000000071727De22E5E9d8BAf0edAc6f37da032` | **EntryPoint v0.7 合约**，跟数据流图一致 |
| 交易状态 | `0x1`（成功） | ✅ |
| gas 消耗 | 270,132 gas | 比普通转账（21,000）大很多——因为同时部署了 Smart Account 合约 |
| 事件日志 | 7 个 log | 包含合约部署、初始化、EntryPoint 记录等 |
| Owner EOA 余额 | 0 ETH | 全程没花 Owner 的钱，**gas 由 Paymaster 赞助** |

### 7 个 Log 的含义

| Log | 合约地址 | 含义 |
|---|---|---|
| log 0-2 | `0x53cf...`（Smart Account） | Smart Account 合约部署 + 初始化事件（Upgraded、OwnershipChanged、Initialized） |
| log 3 | `0x0000...da032`（EntryPoint） | `AccountDeployed` — EntryPoint 记录新账户部署 |
| log 4 | `0x7777...834c`（Paymaster） | Paymaster 赞助 gas 的记录 |
| log 5-6 | `0x0000...da032`（EntryPoint） | `BeforeExecution` + `UserOperationEvent` — 执行记录 |

### 关键发现

1. **Smart Account 地址是预计算的（counterfactual）**：代码里 `smartAccount.address` 在发交易之前就算出来了，因为地址由 owner + salt + factory 确定
2. **第一笔 UserOp 同时部署了合约**：log 0-3 都是部署相关的事件，以后再发 UserOp 就不需要这步了，gas 会少很多
3. **Owner 的 EOA 余额始终是 0**：gas 完全由 Paymaster（`0x7777...`）赞助，这就是 AA 的 gas abstraction
4. **交易发起者是 Bundler，不是我**：链上看到的 `from` 是 Pimlico Bundler，我的 EOA 只负责签名 UserOp，不负责提交交易
5. **整条通路跟数据流图完全吻合**：UserOp → Bundler → EntryPoint → validateUserOp → execute

## Session Key × AI Agent 思考

> 核心问题：AI Agent 要在链上执行操作，Session Key 怎么给它划安全边界？

### 权限维度

- **合约范围**：Agent 只能调用哪些合约？
- **函数范围**：只能调用哪些函数？（如只能 `transfer`，不能 `approve`）
- **金额上限**：单笔限额 / 累计限额
- **时间窗口**：Session Key 多久过期？
- **链范围**：只限 Sepolia？还是跨链？

### 对比传统权限模型

| | 传统 API Key | EOA 私钥 | Session Key |
|---|---|---|---|
| 权限粒度 | 可精细控制（如只读、限具体接口、绑定 IP）| 全权控制（拥有账户所有资产的完全控制权，无法细分）| 可精细控制（限定特定操作、合约、时间窗口、额度等）|
| 撤销方式 | 服务端随时删除或禁用，即时生效| 无法撤销（一旦泄露只能将资产转移到新账户，原私钥永久有效）| 可主动撤销（链上或链下会话失效、过期自动作废、或由授权者吊销）|
| 泄露后果 | 攻击者只能获得该 Key 被授予的有限操作权限，服务端可快速封禁| 账户完全失控，资产可被瞬间全部转走，无法追回| 仅泄露该会话允许的有限操作范围，且可提前撤销或等待自动过期|
| 适合 AI Agent？ | 较适合（服务端可控制权限、定期轮换 Key）| 极不适合（权限过大，泄露风险极高，AI 难以承担）| 非常适合（临时、精细授权，可自动过期，符合 AI 自主操作的安全要求）|

## 卡点 / 踩坑记录

### 1. permissionless.js API 版本不匹配

**问题**：文档示例和实际安装版本的 API 不一致，TypeScript 类型报错。
**解决**：改用 JavaScript 版本，绕过 TypeScript 编译问题。
**教训**：Web3 生态的 SDK 迭代快，文档经常跟不上。跑不通先查版本号和 changelog，不要死磕文档示例。

### 2. RPC 端点连接失败（missing provider）

**问题**：代码中默认的 `https://rpc.sepolia.org` 无法连接。
**解决**：替换为可用的公共 RPC URL（如 publicnode）。
**教训**：公共 RPC 端点不稳定是常态，生产环境应该用 Alchemy / Infura 等付费节点，或至少准备 fallback 列表。

### 3. .env 文件 UTF-8 BOM 导致 dotenv 读取失败

**问题**：Windows 创建的 .env 文件含 BOM（字节顺序标记），dotenv 把 BOM 当成变量名的一部分，导致 API Key 读不到。
**解决**：重新创建无 BOM 的 .env 文件。
**教训**：Windows 环境下创建配置文件注意编码格式。用 VS Code 可以在右下角切换「UTF-8」/「UTF-8 with BOM」。

## 今日收获

### 核心认知

**AA 交易的发起地址不是我的 EOA，而是 EntryPoint 合约地址**——交易通过标准的账户抽象入口处理执行。这是 AA 和普通交易最直观的区别。

普通交易 vs UserOperation 的本质差异：

| 特性 | 普通交易 (Legacy Tx) | 用户操作 (UserOperation) |
|---|---|---|
| 发送主体 | 外部账户 (EOA) | 智能账户 (Smart Account) |
| Gas 支付 | 必须由发起账户支付 | 可由第三方（Paymaster）赞助或用 ERC-20 代币支付 |
| 发送对象 | 直接发送到以太坊主 Mempool | 发送到专门管理 UserOperation 的 Alt Mempool |
| 处理流程 | 由验证者直接打包 | 由 Bundler 收集验证后，批量提交给 EntryPoint 合约处理 |
| 主要作用 | 简单的资产转移或合约交互 | 实现批量交易、Gas 代付、社交恢复等复杂逻辑 |

### 与前几天的连接

- Day 6（RPC 查询）→ Day 8（Indexing）→ **今天（AA）**：从「读链上数据」升级到「让 Agent 安全地写链上数据」
- Indexing 解决的是 Agent 的「眼睛」，AA 解决的是 Agent 的「手」
- 今天实操跑通了完整的 ERC-4337 通路：UserOp → Bundler → EntryPoint → Smart Account，跟数据流图完全吻合

### 工程经验

Web3 开发的坑不在概念，在**环境**：SDK 版本、RPC 端点、文件编码这些「非技术问题」占了调试时间的大头。以后做实操先检查：版本号对不对、网络通不通、配置文件格式对不对。
