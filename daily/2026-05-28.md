## 核心概念

### 1. Web3 Tool Use 解决什么问题

AI Agent 不能直接访问链上数据或发起交易。它需要一个"工具层"来桥接：

```text
用户意图 → Agent 推理 → 决定调用哪个工具 → 工具执行链上操作 → 结果返回 Agent
```

Web3 Tool Use 就是把 RPC 查询、钱包签名、合约调用封装成 Agent 可以理解和调用的标准化工具。每个工具有明确的：
- **输入参数**（地址、合约方法、参数值）
- **输出格式**（查询结果、交易哈希、事件日志）
- **权限边界**（只读还是可写、需要签名还是不需要）

### 2. 四类 Web3 工具

| 类别 | 典型操作 | 是否需要签名 | 风险等级 | 之前实践 |
|---|---|---|---|---|
| RPC 读取 | `eth_getBalance`、`eth_call`、`eth_getLogs` | 不需要 | 低（只读）| Day 6 查 Sepolia 余额和合约状态 |
| 合约读取 | `balanceOf`、`totalSupply`、读取 public 状态变量 | 不需要 | 低（只读）| Day 8 Indexing 场景 |
| 钱包签名 | 签名消息、签名交易、`eth_signTypedData` | 需要私钥/Session Key | 中（不花 gas 但代表授权）| Day 9 UserOp 签名 |
| 合约写入 | `transfer`、`approve`、调用状态变更方法 | 需要签名 + gas | 高（链上不可逆）| Day 9 AA 发 UserOp |

### 3. 工具定义本身也是安全边界

每个工具在定义时就已经划定了 Agent 的能力范围：

- **没有定义转账工具** → Agent 永远不能转账
- **转账工具限制了合约地址参数** → Agent 只能跟指定合约交互
- **工具输出不包含私钥** → Agent 看不到也拿不到私钥

这一点跟 Day 10 的结论直接衔接：安全边界不能只放在 Prompt 上，要放在工具定义和权限模型上。

### 4. 工具调用链：从前几天实践的角度重新理解

**Day 6 的 RPC 查询**，从工具视角看就是：

```
Agent 收到问题："这个地址有多少 ETH？"
  → Agent 选择工具：getBalance(address)
  → 工具执行：调用 Sepolia RPC 的 eth_getBalance
  → 返回结果：0.5 ETH
  → Agent 用这个结果回答用户
```

**Day 9 的 AA UserOp**，从工具视角看就是：

```
Agent 收到指令："向 0x... 转 0.001 ETH"
  → Agent 选择工具：sendUserOperation(to, value)
  → 工具检查：金额在限额内？目标在白名单？
  → 工具执行：构造 UserOp、用 Session Key 签名、提交到 Bundler
  → 返回结果：UserOp hash
  → Agent 回复用户："已提交，哈希: 0x..."
```

## 今日收获

今天最重要的理解：

**Web3 Tool Use 不是"让 Agent 能调 API"，而是"用工具定义画一条安全边界"。**

每个工具的定义（能调什么合约、能传什么参数、需要什么权限）本身就是一层约束。Agent 只能调用你定义过的工具，只能传工具允许的参数范围。这意味着：

- 如果你没定义 `approve` 工具，Agent 永远不能做授权操作
- 如果你在 `transfer` 工具里写死了合约地址和金额上限，Agent 突破不了这个限制
- Prompt Injection 最多诱导 Agent "想"做坏事，但工具层不给他这个能力

这跟 Day 10 的结论完全一致：安全不在 Prompt 里，在权限模型和工具约束里。
