## 核心收获

### 大语言模型（LLM）

LLM 是概率模型，不是事实数据库。它生成的是"概率上合理的输出"，不是可信事实。

关键判断：**模型输出是候选结果，不是事实本身。**

- Token：模型处理文本的基本单位，影响上下文容量和成本
- Embedding：把文本映射成向量，用于语义搜索和 RAG
- Transformer：擅长在上下文里找模式，但没有稳定数据库
- Hallucination（幻觉）：模型会编造看似合理但不真实的内容，越靠近真实执行越危险

在 AI × Web3 中：LLM 负责理解和生成层，真正执行还需要数据层、编排层、执行层、安全层配合。

---

### 提示词（Prompt）

Prompt 是任务接口设计，不是魔法咒语。它把模糊任务变成模型可稳定执行的工作说明。

关键判断：**Prompt 是软约束，不是安全边界。真正的边界由代码、权限、校验和审计承担。**

- Instruction：角色 + 任务 + 禁止行为 + 输出格式
- Few-shot：放少量示例让模型模仿，但示例要跟 eval 一起维护
- Structured Output：让模型返回 JSON 等固定结构，方便后续代码处理
- Prompt Injection：攻击者通过输入让模型忽略原始规则，Agent 场景尤其危险

---

### 上下文（Context）

Context 是模型这一次能看到的信息空间。问题不是"塞多少"，而是"分清楚"。

关键判断：**模型只能基于它看见的上下文行动；系统必须决定什么能进上下文、带着什么身份进去。**

- Context Window：模型一次能处理的最大范围，大不等于用得好
- Context Engineering：设计信息进入模型的方式，分层放置（指令层、任务层、事实层、知识层、记忆层）
- Memory：跨请求保留的信息，不能替代实时授权
- Knowledge Base：外部知识库，需要维护来源、版本、更新时间

---

### 检索增强生成（RAG）

RAG 是把外部知识取回、筛选、引用、交给模型使用的证据链，用来减少过期知识和无来源回答。

关键判断：**RAG 的可靠性取决于证据链，不取决于向量库这个名词。**

- Chunking：按结构切文档（标题、函数说明、FAQ），每个 chunk 保留来源和版本
- Vector DB：存 embedding 并按相似度检索，但向量相似 ≠ 答案正确
- Retriever：取回候选材料，需要结合 metadata filter，不能只看语义相似度
- Rerank：对候选结果重新排序，减少"标题相似但内容不对"的情况
- Citation：答案必须能追溯到来源，否则用户无法验证

---

### 智能体（Agent）

Agent 是能围绕目标持续调用工具、读取状态、调整步骤的 AI 系统。

关键判断：**Agent 不是自主权本身，而是被约束的执行循环。目标、工具、状态、权限和停止条件缺一不可。**

- Tool Use：让 Agent 从"会回答"变成"能做事"，但工具让错误从文本变成真实动作
- Planning：模型生成的计划是候选路线，不是授权
- State：任务状态必须外置可查，不能只藏在 prompt 历史里
- Reflection：自我检查可以提高质量，但不能替代外部验证
- Multi-Agent：分工协作，但要防止上下文传递丢失和责任边界模糊

---

### 框架（Frameworks）

框架解决工程组织问题，不替你定义产品价值，也不保证模型输出正确。

关键判断：**先理解工作流，再决定用不用框架。**

- LangChain：组件库，适合快速组合能力，小心"抽象太早"
- LangGraph：状态机，适合多轮工具调用、重试、人工确认、分支的复杂流程
- OpenAI Agents SDK：工具、handoff、guardrails、tracing 的组合结构
- DSPy：把 prompt pipeline 写成可优化程序，需要明确数据集和评估指标

---

### 氛围编程（Vibe Coding）

Vibe Coding 是人和 AI Coding Agent 共同迭代软件的工作方式：人负责方向、约束和验收，Agent 负责生成、修改和执行。

关键判断：**AI 能帮你写更多代码，但不能替你承担工程判断。**

- 任务要小：越具体越有边界，Agent 越容易产出可审查结果
- 上下文要准：让 Agent 看对文件比写长篇需求更重要
- 验证要硬：运行测试、类型检查、构建，比"看起来对"可靠
- 链上代码风险更高：合约、签名、权限、支付脚本不能只靠 Agent 生成后直接上线

---

### 模型上下文协议（MCP）

MCP 把模型和外部工具、数据源之间的连接标准化，解决的是"如何以可描述、可复用、可管理的方式使用外部能力"。

关键判断：**模型不应该直接拥有世界；它应该通过明确协议访问被授权的上下文和工具。**

- Server：暴露 resources、tools、prompts，设计重点是边界
- Client：连接模型和 server，负责用户确认、权限提示、会话隔离
- Tool Schema：描述工具名字、参数、返回值和约束，写得模糊模型就会用错误参数填空
- Permission：读文档和发支付的风险完全不同，MCP 方便不等于安全

---

### 评估（Evaluation）

Evaluation 是把"感觉效果不错"变成"系统可持续改进"的方法。

关键判断：**不能被重复测量的 AI 行为，就不能被稳定改进。**

- Harness：运行 eval 的框架，关键是可重复
- Golden Set：30-100 条高质量样本，覆盖正常问题、边界问题、历史 bug
- LLM-as-Judge：用模型给模型评分，适合开放式答案，但不能神化
- Regression：防止旧问题复发，每次改动前后都跑一遍
- Observability：线上观察系统行为，记录输入、检索结果、工具调用、错误和成本

---

### 微调（Fine-tuning）

Fine-tuning 让模型更稳定地学习某种格式、风格或行为模式，但不适合补实时知识或修权限边界。

关键判断：**先有 eval，再谈 fine-tuning。先修数据，再修模型。**

- SFT（监督微调）：用输入输出样本让模型学习任务回答方式，对数据质量非常敏感
- LoRA：参数高效微调，降低训练成本，但不是魔法
- Dataset：训练集、验证集、测试集、回归集要分开，不能把测试集拿去训练
- Overfitting：模型把训练数据记太死，真实用户一问就崩

---

### 推理服务（Inference）

Inference 决定模型能力如何被产品消费：延迟、成本、上下文、稳定性和部署边界的综合选择。

关键判断：**推理服务的核心不是"跑出答案"，而是在约束条件下交付可用答案。**

- API Model：上手快，但仍要处理速率限制、超时、重试、日志脱敏、账单控制
- Local Model：隐私可控、成本可控，但受硬件、显存、量化方式限制
- Quantization：降低精度换速度和体积，可能影响输出质量，要用真实任务测试
- Serving：并发、队列、流式输出、缓存、监控、失败处理和版本管理

## 概念关系图（用自己的话）

```
用户目标
  ↓
Prompt（任务接口）
  ↓
Context（信息空间）← RAG（外部知识检索）← Knowledge Base
  ↓
LLM（推理生成）← Fine-tuning（行为调整）← Inference（部署服务）
  ↓
Agent（执行循环）← Frameworks（工程组织）← MCP（工具协议）
  ↓
Tool Use（外部动作）
  ↓
Evaluation（持续改进）← Vibe Coding（开发方式）
```
