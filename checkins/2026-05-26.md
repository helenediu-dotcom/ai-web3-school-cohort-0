## 核心收获

### 1. AA 交易的 from 不是我的 EOA

链上验证发现：交易 `from` 是 Pimlico Bundler（`0x4337...`），`to` 是 EntryPoint v0.7（`0x0000...da032`）。我的 EOA 只负责签名 UserOp，不负责提交交易，也不需要有余额。

### 2. 普通交易 vs UserOperation

| 特性 | 普通交易 (Legacy Tx) | 用户操作 (UserOperation) |
|---|---|---|
| 发送主体 | 外部账户 (EOA) | 智能账户 (Smart Account) |
| Gas 支付 | 必须由发起账户支付 | 可由第三方（Paymaster）赞助或用 ERC-20 代币支付 |
| 发送对象 | 直接发送到以太坊主 Mempool | 发送到专门管理 UserOperation 的 Alt Mempool |
| 处理流程 | 由验证者直接打包 | 由 Bundler 收集验证后，批量提交给 EntryPoint 合约处理 |
| 主要作用 | 简单的资产转移或合约交互 | 实现批量交易、Gas 代付、社交恢复等复杂逻辑 |

### 3. 第一笔 UserOp 同时部署 Smart Account

gas 消耗 270k（普通转账只要 21k），因为第一笔交易同时部署了 Smart Account 合约。链上 7 个 log 中前 4 个都是部署相关事件。后续 UserOp 不需要再部署，gas 会少很多。

### 4. Session Key 是 AI Agent 上链的安全模型

| | API Key | EOA 私钥 | Session Key |
|---|---|---|---|
| 权限粒度 | 可精细控制 | 全权控制，无法细分 | 可精细控制（合约/方法/额度/时效） |
| 泄露后果 | 有限，可快速封禁 | 账户完全失控 | 仅限该会话范围，可撤销或自动过期 |
| 适合 AI Agent | 较适合 | 极不适合 | 非常适合 |

## 踩坑记录

1. **permissionless.js API 版本不匹配**：文档示例和实际安装版本 API 不一致 → 改用 JavaScript 版本绕过 TypeScript 编译问题
2. **RPC 端点失效（missing provider）**：`rpc.sepolia.org` 无法连接 → 替换为可用的公共 RPC
3. **.env 文件 UTF-8 BOM**：Windows 创建的 .env 含 BOM，dotenv 无法正确识别变量名 → 重新创建无 BOM 的 .env

**教训**：Web3 开发的坑不在概念，在环境——SDK 版本、RPC 端点、文件编码这些「非技术问题」占了调试时间的大头。

## 与 AI×Web3 的连接

**Day 6（RPC 查询）→ Day 8（Indexing）→ Day 9（AA）**：从「读链上数据」升级到「让 Agent 安全地写链上数据」。
