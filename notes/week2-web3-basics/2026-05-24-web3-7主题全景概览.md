## 1. Dev Stack（开发栈）

**定义**：从写合约 → 测试 → 部署 → 前端调用 → 浏览器验证的整条工程链路。

**关键概念**：
- **Remix**：浏览器 IDE，适合实验和入门，不适合长期协作
- **Hardhat**：JS/TS 工程框架，适合和前端 + CI 集成
- **Foundry**（forge/cast/anvil）：Solidity-native 命令行，跑测试快，支持 fuzz/fork test
- **OpenZeppelin**：标准合约库（ERC-20/721、AccessControl、Ownable、Pausable），但「用了库 ≠ 安全」
- **viem / wagmi**：前端读写链，wagmi 提供 React hooks

**为什么重要**：链上交易不可逆，开发工具的核心价值是「把错误前移」——本地能复现、地址 ABI 版本化、测试覆盖权限边界。

**AI×Web3 位置**：Agent 可以跑 `forge test`、生成部署脚本、解释 ABI，但高风险命令必须有人工确认门。

**一句话**：工具链越清楚，链上执行越可控。

## 2. Network（网络）

**定义**：交易能否被打包、状态能否被同步、费用如何产生、L2/L1 如何分工的基础环境。

**关键概念**：
- **Block**：交易批量提交单位，有顺序、有 gas limit、引用前一区块
- **Consensus**：让无中心节点对状态达成一致（影响 confirmation 数和 reorg）
- **PoS**：以太坊当前共识；验证者质押 ETH 出块，错误会被 slashing
- **Testnet**：流程验证（如 Sepolia），不能证明经济安全
- **L2 / Rollup**：把执行搬下来、把证明/数据放回 L1；optimistic vs zk

**为什么重要**：很多看起来是钱包/前端的问题，根源是网络——切错链、RPC 延迟、L2 提现等待。

**AI×Web3 位置**：Agent 必须知道自己在哪条链上操作（chain id、RPC 来源、合约地址），不能让模型「猜」。

**一句话**：链上不是写数据库，而是在共识维护的状态机上提交请求。

## 3. Account Abstraction（账户抽象）

**定义**：把账户的验证逻辑、付 gas 方式、权限策略从固定的 EOA 里解放出来，让账户变成可编程合约。

**关键概念**：
- **ERC-4337**：用 `UserOperation` + `EntryPoint` + `Bundler` 的标准流程，不改 EVM 共识层
- **Smart Account**：合约控制的账户，支持多签 / 社交恢复 / 批量交易
- **Bundler**：收集 UserOperation 并提交到 EntryPoint，类似交易打包器
- **Paymaster**：替用户付 gas（新用户 onboarding / 用 USDC 付 gas）
- **Session Key**：给应用或 Agent 的**临时受限权限**（指定合约 + 额度 + 时效 + 链）

**为什么重要**：核心不是「免 gas」，而是把权限从「有/没有私钥」升级为「在什么条件下允许什么动作」。

**AI×Web3 位置**：**这是 Agent Wallet 的底座**。Session Key 让 Agent 可以在受限范围内自动执行，高风险动作仍需用户签名。

**一句话**：账户抽象不是让 AI 更自由，而是让 AI 的自由被规则包起来。

## 4. DeFi（去中心化金融）

**定义**：建立在智能合约上的开放金融系统——资产、交易、借贷、稳定币、流动性被协议化组合。

**关键概念**：
- **Token**（ERC-20）：要看合约地址、decimals、增发权限、是否可暂停
- **AMM**：流动性池 + 定价公式替代订单簿；关注滑点、价格影响、MEV（sandwich）
- **Lending**：抵押率、健康因子、清算价、价格预言机依赖
- **Stablecoin**：法币储备 / 超额抵押 / 算法机制，「稳定」来自机制不是保证
- **Liquidity**：有价格 ≠ 能成交；流动性决定可执行性

**为什么重要**：DeFi 难点不是「合约能不能跑」，而是**资产和机制如何相互影响**——一个 swap 背后藏着 AMM 曲线、滑点、MEV、预言机。

**AI×Web3 位置**：AI 适合做信息整理（解释交易、监控仓位、清算预警），自动执行需要白名单 + 滑点上限 + 模拟 + 人工确认。

**一句话**：DeFi 把金融规则变成可组合、可验证、也可被攻击的链上协议。

## 5. Oracle（预言机）

**定义**：把链外信息（价格、随机数、储备、事件结果）以合约可信任的方式带进链上。

**关键概念**：
- **Price Feed**：最常见的 oracle 形式（如 Chainlink ETH/USD），看 feed 地址、decimals、更新时间
- **Data Feed**：不只价格，还有储备证明、利率、事件结果（RWA / 保险）
- **Oracle Risk**：数据源被操纵、更新延迟、低流动性资产价格被攻击、stale price 没检查
- **AI Oracle**：把模型推理（评分/分类/判断）作为 oracle 输出——必须记录输入、模型版本、prompt，并设计争议处理

**为什么重要**：oracle 是链上系统的**信任边界**。一个错误 feed 会引发错误清算 → 坏账 → 套利攻击。

**AI×Web3 位置**：**AI Oracle 是关键交叉点**。但「模型说 → 合约执行」很危险，应该走「记录来源 + 置信度 + 高风险场景 human-in-the-loop」。

**一句话**：预言机不是「真实世界 API」，而是合约愿意信任的一套数据提交和验证机制。

## 6. Indexing（索引）

**定义**：把链上的区块/交易/事件/合约状态整理成产品和 AI Agent 能快速查询的结构化数据。

**关键概念**：
- **Event Indexing**：监听合约 event（Transfer / OrderCreated / Deposit）转成数据库表
- **Subgraph**（The Graph）：声明式定义「监听什么 + 怎么 mapping + GraphQL schema」
- **RPC**：链状态接口，**不是索引服务**——大量历史查询会被 rate limit
- **Data Pipeline**：RPC → event listener → 解码 ABI → DB → 处理 reorg → API/GraphQL → dashboard/Agent context

**为什么重要**：链上是事实来源，**索引层是可用数据层**。没有它，前端做不出用户历史/排行榜，AI Agent 拿不到稳定上下文。

**AI×Web3 位置**：Agent 的链上上下文几乎都来自索引层。索引过期或解码错误，模型推理再强都是错的事实。

**一句话**：合约 event 是产品和 AI 的数据入口——设计合约时就要为索引设计。

## 7. Security（安全）

**定义**：从合约设计 → 权限 → 测试 → 模拟 → 监控 → 应急的一整套工程流程，不是上线前审一次就完。

**关键概念**：
- **Reentrancy**：外部调用未完成前被再次调用 → Checks-Effects-Interactions / ReentrancyGuard
- **Access Control**：mint/burn/pause/upgrade/withdraw 的权限——owner 是 EOA 还是多签？有 timelock 吗？
- **Audit**：是发现问题，不是安全保证书；要看 commit 一致性和接受风险项
- **Simulation**：交易签名前预演（Tenderly），挡住合约地址错、approve 异常、滑点过大
- **Monitoring**：大额转账 / 权限变更 / 升级 / 价格异常 → 监控必须配响应链路（pause/告警/恢复）

**为什么重要**：合约公开 + 资金公开 + 攻击者可反复模拟，错误成本远高于普通应用。

**AI×Web3 位置**：AI 让安全边界更复杂——prompt injection、错误工具调用、危险交易生成。设计原则：**模型建议 → 工具返回事实 → policy 限权 → simulation 预演 → human check → monitoring 记录**。

**一句话**：安全的核心不是「没 bug」，而是把可预见风险挡在执行前，并在执行后能快速止损。
